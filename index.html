Projects

Topaz

Topaz is a multimedia engine written in C++. Its features include but are not limited to: Heavy-duty 3D rendering, lightweight 2D (gui) rendering, a minimalistic sound system but with the key advantage over other engines being its extreme independence.

Topaz is 'independent' in the way that once you begin using it, the internal underlying system stays out of your way whilst being easily accessible, allowing you to have maximum control over your project whilst making Topaz incredibly easy to modify for your own use. Topaz also fanatically adheres to the C++ ISO Standard. This means that the source code for Topaz is very portable, being able to run on all modern platforms that the C++ ISO Standard allows.

The primary design choice for Topaz is minimalism. So many other multimedia engines are incredibly bloated full of features that are largely unnecessary in my own opinion. I designed Topaz to the best of my ability to be the antonym to verbosity. This being said, Topaz is a large project, so remains realistically minimalistic. For example, measures are taken to prevent the use of the preprocessor as much as possible, when it could be used to make the engine compact at the cost of readability.

Asteroids

Asteroids is a very simple game designed to be a more modern adaptation of Atari's popular arcade game of the same name from the 1970s. This incarnation of Asteroids is now 3D, and has an exponential difficulty system. This is different to a normal difficulty system such that at the beginning of a level, gameplay is very slow and easy indeed. Once the end of the current level is approached, the difficulty is dramatically higher than before, despite being on the same level. Higher levels increase the minimum difficulty and maximise the maximum difficulty whilst ensuring that the later stages of early levels aren't unrealistically difficult.

The graphics, sound system and the program as a whole are deliberately very minimal. Asteroids uses a very early version of Topaz, so modding/forking from it is discouraged. Essentially, this project remains here as a reminder of the past, and my journey as a programmer. It's also a fun way for me to kill half an hour when my internet is down.

MDL

MDL stands for Minimalist Data Language. MDL aims to provide features that technologies such as XML or JSON can provide, whilst remaining minimalistic and avoiding feature bloat as much as possible. To describe MDL very briefly, I would call it a more lightweight alternative to YAML.

MDL also acts somewhat uniquely when loading a MDL file. The interpreter will not attempt to analyse the file in small chunks like most would. In fact, it slurps the entirety of the data-file at once (Assuming the file is not so large as to necessitate fragmentation of data), and then organises all the raw data into highly-accessible MDL buffers and helper functions upon initialisation.

This means that MDL does the vast majority of its processing upon initialisation, and creating/deleting/reading tags and sequences becomes almost instantaneous. The major drawback to this feature is that when creating and editing tags, the MDL buffers must be re-parsed if you wish to read changes made previously. However, work is currently being done to alleviate this issue to an extent, applying edits to not only the data-file, but also the MDL buffers associated with that file, such that a re-parse is unnecessary.
